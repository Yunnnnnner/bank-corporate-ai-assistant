### 3.2 需新增的核心能力

#### 能力1：重排序（Re-ranking）

**问题**：RRF混合检索召回的结果，相关性排序不一定最优

**解决方案**：在召回后增加交叉编码器重排序

```python
# 新增 retrieval/reranker.py
from sentence_transformers import CrossEncoder

class Reranker:
    def __init__(self):
        # 中文重排序模型
        self.model = CrossEncoder('BAAI/bge-reranker-large')

    def rerank(self, query: str, contexts: List[RetrievedContext]) -> List[RetrievedContext]:
        pairs = [(query, ctx.text) for ctx in contexts]
        scores = self.model.predict(pairs)
        for ctx, score in zip(contexts, scores):
            ctx.score = float(score)
        return sorted(contexts, key=lambda x: x.score, reverse=True)
```

**效果**：准确率提升约10-15%（尤其对长文档）

#### 能力2：查询改写（Query Rewriting）

**问题**：用户可能用口语/缩写提问，降低召回质量

**解决方案**：调用Claude对查询进行改写扩展

```python
# 在 retrieval/retriever.py 增强
def rewrite_query(self, original_query: str) -> List[str]:
    """生成多个查询变体，扩大召回覆盖"""
    prompt = f"""你是一位银行信贷专家。请将以下用户问题改写为3个不同角度的精确查询，
    以便在银行信贷规则知识库中检索相关内容。
    
    用户问题：{original_query}
    
    输出格式：每行一个改写版本，仅输出改写后的问题，不要编号。"""
    # 调用Claude生成查询变体
    variants = call_claude(prompt).strip().split('\n')
    return [original_query] + variants[:2]   # 原始 + 2个变体
```

#### 能力3：多路召回合并（Multi-query Retrieval）

```python
def multi_query_retrieve(self, question: str) -> List[RetrievedContext]:
    """对多个查询变体分别检索，合并去重排序"""
    queries = self.rewrite_query(question)
    all_results = {}
    for q in queries:
        results = self.retrieve(q)
        for r in results:
            if r.chunk_id not in all_results:
                all_results[r.chunk_id] = r
            else:
                # 取最高分
                all_results[r.chunk_id].score = max(
                    all_results[r.chunk_id].score, r.score
                )
    return sorted(all_results.values(), key=lambda x: x.score, reverse=True)[:self.top_k]
```

#### 能力4：上下文压缩（Context Compression）

**问题**：召回的片段可能包含大量不相关内容，浪费 Token

**解决方案**：在注入 Prompt 前先提取相关句子

```python
class ContextCompressor:
    def compress(self, query: str, context: str) -> str:
        """从上下文中提取与问题最相关的部分"""
        prompt = f"""从以下文本中提取与问题"{query}"直接相关的句子，
        保持原文不变，删除无关内容：\n{context}"""
        return call_claude(prompt)
```

## 四、L3 增强推理层技术能力

### 4.1 结构化数据联合查询（Text-to-SQL）

**场景**：用户问参数类问题，需要精确数字

```
用户："中型企业的最高授信额度是多少？"

传统RAG：检索文档片段，可能找到也可能找不到
Text-to-SQL：
  1. 识别为参数查询
  2. 生成SQL：SELECT max_value FROM product_rules 
               WHERE product_code='loan' 
               AND rule_type='额度'
               AND customer_type='中型企业'
  3. 返回精确结果：2亿元（来源：[文件名]）
```

**实现**

```python
# 新增 retrieval/sql_retriever.py
class SQLRetriever:
    def __init__(self, meta_store: MetaStore):
        self.ms = meta_store

    def is_parameter_query(self, question: str) -> bool:
        """判断是否为需要精确数值的查询"""
        keywords = ['多少', '上限', '下限', '最高', '最低', '不超过',
                    '不低于', '比例', '期限', '利率', '几年', '几个月']
        return any(kw in question for kw in keywords)

    def lookup_parameter(self, question: str) -> Optional[str]:
        # 调用Claude生成SQL，执行后返回结果
        sql = self._generate_sql(question)
        return self.ms.execute_query(sql)
```

## 七、银行数据安全合规要求

### 7.1 数据分级保护

| 数据级别 | 内容 | 处理要求 |
|---------|------|---------|
| 公开 | 监管公开文件、产品说明书 | 可使用外部API |
| 内部 | 内部操作规程、业务规则 | 建议本地部署LLM |
| 机密 | 客户信息、审批数据、定价参数 | 必须本地部署，禁止外传 |
| 绝密 | 集中授信底线、风险偏好设置 | 隔离系统，审计追溯 |

### 7.2 部署安全要求

**外部API使用限制（建议）**
- Claude/OpenAI API：仅用于公开文件和内部非敏感规则
- 客户相关查询：必须走私有化部署的本地模型
- 所有外部API调用须记录日志，定期审计

**本地化部署方案（满足监管合规）**
```
私有化部署方案：
  Embedding: BAAI/bge-large-zh（本地）
  LLM: Qwen2.5-72B 或 DeepSeek-V3（本地）
  向量库: Milvus（本地服务器）
  数据: 加密存储 + 访问控制
  网络: 内网隔离，不出外网
```
