1产品概述

1.1 一句话定义

面向银行公司业务条线的智能合规助手，把分散在 10+ 份业务规则、内部政策、监管文件、法律法规中的碎片化知识，通过 Wiki-RAG 技术融合为可推理、可引用、可审计的合规决策能力。

1.2 核心价值

价值维度描述
效率客户经理一笔预审从 30-60 分钟 → 2-3 秒，效率提升 600 倍
准确单笔预审跨文件命中 5-8 份规则，杜绝漏审 / 错审
合规每条结论必带文件名 + 原文引用，全程留痕可审计
时效监管新规当天入库即生效，无需重训模型
私有行内审批权限 / 定价底线 / 内部流程 注入到每次推理

1.3 关键差异（与普通 RAG）

本产品采用 Wiki-RAG 架构 —— 把规则建模为「有命名空间 / 有层级 / 有引用 / 有版本 / 有权限」的知识网络，而非扁平片段集合。

2背景与目标

2.1 业务痛点

痛点当前现状影响
规则散落10+ 份文档分散在 OA / 共享盘 / 法务系统客户经理逐份翻查
表述不一"小微 / 普惠 / 中小微" 同义不同名关键词搜索召回不全
时效混乱旧规未废止、新规未公示并存审批引用过期条款
综合判断难一笔申请涉及准入/期限/担保/定价/审批 5+ 维度业务员易漏审，分行打回
合规审计弱LLM 直接答复无引用 → 不可追溯监管检查存在风险

2.2 商业目标（OKR）

O：把公司业务条线的合规预审从"经验依赖"升级为"知识库驱动"

KR衡量指标目标值
KR1客户经理预审 TAT（Turn-Around Time）≤ 5 秒
KR2单笔预审规则覆盖率≥ 92%
KR3因规则遗漏导致的二次返工率≤ 3%
KR4合规官人工抽查通过率≥ 95%
KR5月活用户（客户经理 + 审批官）≥ 800 人

2.3 不做什么（Out of Scope）

  • ❌ 不替代正式审批（仅作预审参考）
  • ❌ 不接管客户档案管理（仍走核心系统）
  • ❌ 不直接落账、不发起放款
  • ❌ 不处理零售个人贷款（本期仅公司业务）

4产品定位

4.1 Wiki-RAG vs 普通 RAG · 立标对比

能力维度普通 RAGWiki-RAG（本产品）
文档建模扁平片段集合命名空间 + 类目层级（业务/政策/监管/法规 4 大类）
切分策略固定窗口滑动切分语义感知：按"第X条/第X章/(一)(二)"切分
检索方式纯向量相似度混合检索（语义 α=0.6 + BM25 0.4）+ RRF 融合
知识关联片段彼此独立同义词桥接 + 跨文档引用 + 实体追踪

引用溯源无引用每条结论必带文件名 + 原文关键句
对话记忆无状态5 层记忆（窗口/摘要/实体/引用/偏好）
知识更新重建全量索引增量入库（SHA256 去重）+ 软删除
合规审计无法追溯全程留痕
权限管控统一暴露RBAC 按文档分级

4.2 一句话区分

普通 RAG：把文档塞进向量库，检索相似句子。

Wiki-RAG（本产品）：把规则建模成知识网络，检索"有命名空间 / 有层级 / 有引用 / 有版本 / 有权限"的知识节点，并具备跨轮记忆。

6详细功能需求

6.1 场景 1：信贷预审合规助手（P0）

6.1.1 功能描述

客户经理在表单中录入企业 + 业务要素，系统并行启动 8 个维度的 RAG 检索，2-5 秒内生成结构化预审报告。

6.1.2 输入字段（13 项）

字段类型必填说明
企业名称文本✅
成立年份数字✅用于推算经营年限
所属行业文本✅触发行业准入查询
企业类型枚举✅小微/中型/大型
年营业收入(万)数字✅
资产负债率(%)数字✅触发财务准入查询
内部评级枚举✅AAA/AA/A/BBB/BB
不良记录布尔✅
产品类型枚举✅流贷/固贷/并购/票据/ABS
申请金额(万)数字✅触发权限查询
期限(月)数字✅
贷款用途文本✅触发用途合规检查
担保方式 + 估值复合✅触发质押率检查

6.1.3 处理流程

1. 构造 8 维检索查询（RETRIEVAL_QUERIES 模板填充）
2. 每条查询经 QueryRewriter 派生 2 个变体 → 24 次检索
3. 并行执行混合检索（ChromaDB + SQLite FTS5）
4. 按 chunk_id 合并去重，取 max(score)，保留 top-12
5. 拼装 system prompt + 申请描述 + 检索内容
6. Claude Sonnet 4.6 流式生成 8 段式报告
7. 解析结论（PASS/CONDITIONAL/REJECT），抽取来源文件
8. 渲染：检索动画 + 命中文件表 + 报告 + Verdict 卡片

6.1.4 输出格式（8 段式报告）

  1. 一、基本准入审查
  2. 二、额度与期限合规性
  3. 三、担保合规性
  4. 四、定价建议
  5. 五、审批权限建议
  6. 六、风险提示
  7. 七、预审结论（PASS / CONDITIONAL / REJECT）
  8. 八、所需材料清单

每条结论格式：✅/⚠️/❌ [条件名]：[判断]（依据：[文件名] - [原文关键句]）

6.1.5 性能要求

  • 端到端 ≤ 5 秒（P95）
  • 单笔命中 ≥ 5 份规则文档
  • 引用召回率 ≥ 95%

6.2 场景 2：智能咨询多轮对话（P0）

6.2.1 功能描述

客户经理可以连续提问，系统通过 5 层记忆机制实现指代消解、上下文延续、实体追踪。

6.2.2 5 层记忆机制

层级范围触发条件实现
① 短期窗口最近 10 轮原文始终启用sliding window
② 长期摘要> 10 轮的对话压缩turn ≥ 20LLM summarize → store
③ 实体追踪企业名 / 金额 / 产品等始终启用NER + persistent kv
④ 引用记忆已被引用的文档列表始终启用doc_id set
⑤ 偏好记忆角色身份 / 详略风格用户主动声明user_pref dict

6.2.3 关键功能点

  • 指代消解："那 800 万呢" → 自动补全企业类型
  • 接续问答："那利率怎么算" → 复用上文实体
  • 摘要提示：「请汇总一下」→ 调用所有引用记忆
  • 实时可视化：右侧记忆面板展示当前 5 层状态

6.2.4 接口

  • POST /api/chat { session_id, query }
  • GET /api/chat/memory/{session_id} 返回当前记忆状态
  • DELETE /api/chat/{session_id} 清空会话

6.3 场景 3：产品方案智能比对（P1）

6.3.1 处理流程

1. LLM 解析诉求 → 提取关键信号（场景/期限/金额/担保/期望）
2. 按关键信号匹配产品候选池（保理/质押贷/e销贷/票据贴现/...）
3. 并行检索每个产品的 8 维要素（额度/期限/利率/审批/材料/...）
4. 按场景适配度打分，标注每行最优值
5. LLM 生成综合推荐 + 引用溯源

6.3.2 对比维度（8 列）

融资额度 / 融资期限 / 利率 / 审批层级 / 所需材料 / 行内权限 / 适用条件 / 风险点

6.4 知识库管理（P0）

6.4.1 知识库总览页

  • 4 项关键统计：文档总数 / 文本片段 / 向量索引 / 索引大小
  • 文档清单按 4 大类目分组：业务规则 / 内部政策 / 监管文件 / 法律法规
  • 每份文档显示：chunk 数 / 章节数 / 关键主题

6.4.2 摄入流水线

  • 6 步处理：扫描加载 → 格式解析 → 数据清洗 → 语义分块 → 向量化 → 双库存储
  • 支持格式：PDF / DOCX / MD / XLSX / CSV
  • 增量摄入：SHA256 去重，已索引的文档自动跳过

6.5 RAG 知识库日志（P0 · 新增）

6.5.1 文件状态机

            ┌──[审核拒绝]──→[退回]
            │
[草稿]──→[审核中]──→[生效中]──┬──→[已废止]（被新版替代 / 主动废止）
                                │
                                └──→[暂停使用]（被临时文件覆盖）
                                       │
                                       └──→[恢复生效]（临时文件到期）

6.5.2 元数据字段（每份文档必填）

字段类型说明

类型enum常规 / 临时
分类enum业务规则 / 内部政策 / 监管文件 / 法律法规
状态enum生效 / 暂停 / 废止
入库时间datetime
发布部门 / 发布人str

6.5.3 临时文件 → 常规文件覆盖机制（关键）

业务规则：

  1. 临时文件入库时声明 覆盖文件 ID
  2. 系统将被覆盖的常规文件状态置为「暂停使用」
  3. 检索时优先返回临时文件片段
  4. 临时文件到期（定时任务每天 00:00 检查）：
    • 状态 → 已废止，废止日期 = 到期日期
    • 同步将被覆盖的常规文件恢复「生效」
  5. 全程留痕，操作记录写入变更日志

示例：《阶段性提高小微企业流贷单户限额的临时通知（2026年第3号）》

· 类型：临时

· 有效期：2026-04-01 ~ 2026-06-30

· 覆盖：《对公信贷产品完整手册》§1.4 小微限额条款

· 后果：该手册状态变为「暂停使用」+ 红色删除线

6.5.4 UI 要求

  • 顶部统计卡：生效中 / 暂停中 / 已废止 / 临时文件数（含倒计时）
  • 临时文件区独立成表，金边高亮置顶，含有效期进度条
  • 常规文件清单：被暂停的文件文件名加红色删除线 + "暂停使用"标签
  • 已废止文件：红背景 + 删除线 + 半透明
  • 底部变更日志：按时间倒序，含预定的自动化操作

10.2 交互原则

  1. 即点即响：核心动作 ≤ 1 次点击触发
  2. 过程可见：检索 / 推理过程实时动画展示
  3. 引用可点：每条引用可点击展开原文（M3 阶段）
  4. 状态明确：颜色 + 图标 + 文字 三重提示
  5. 可逆操作：所有删除 / 废止可在 30 天内回滚

10.3 关键视觉处理

  • 被废止文件：文件名 red strikethrough
  • 被暂停文件：删除线 + 黄色背景行 + "暂停使用" pill
  • 临时文件：金色边框 + 进度条倒计时
  • 生效中：绿色 pill + 文字状态

13风险与依赖

13.1 技术风险

风险等级缓解措施
LLM 幻觉🔴 高强约束 prompt + 引用强制 + 标注集回归

风险等级缓解措施
数据出域🔴 高M6 私有化部署
引用错误导致违规🔴 高双人复核 + 月度抽查
临时文件未及时入库🟡 中与法务系统对接，自动同步
审计追溯不全🟡 中全程留痕 + 30 天回溯
