## 1. 产品概述

### 1.1 一句话定义
**面向银行公司业务条线的智能合规助手**，把分散在 10+ 份业务规则、内部政策、监管文件、法律法规中的碎片化知识，通过 **Wiki-RAG** 技术融合为可推理、可引用、可审计的合规决策能力。

### 1.2 核心价值
| 价值维度 | 描述 |
|---|---|
| **效率** | 客户经理一笔预审从 30-60 分钟 → 2-3 秒，效率提升 600 倍 |
| **准确** | 单笔预审跨文件命中 5-8 份规则，杜绝漏审 / 错审 |
| **合规** | 每条结论必带文件名 + 原文引用，全程留痕可审计 |
| **时效** | 监管新规当天入库即生效，无需重训模型 |
| **私有** | 行内审批权限 / 定价底线 / 内部流程 注入到每次推理 |

### 1.3 关键差异（与普通 RAG）
本产品采用 **Wiki-RAG** 架构 —— 把规则建模为「<u>有命名空间 / 有层级 / 有引用 / 有版本 / 有权限</u>」的知识网络，而非扁平片段集合。详见 [§4 产品定位](#4-产品定位)。

## 2. 背景与目标

### 2.1 业务痛点

| 痛点 | 当前现状 | 影响 |
|---|---|---|
| 规则散落 | 10+ 份文档分散在 OA / 共享盘 / 法务系统 | 客户经理逐份翻查 |
| 表述不一 | "小微 / 普惠 / 中小微" 同义不同名 | 关键词搜索召回不全 |
| 时效混乱 | 旧规未废止、新规未公示并存 | 审批引用过期条款 |
| 综合判断难 | 一笔申请涉及准入/期限/担保/定价/审批 5+ 维度 | 业务员易漏审，分行打回 |
| 合规审计弱 | LLM 直接答复无引用 → 不可追溯 | 监管检查存在风险 |

### 2.2 商业目标（OKR）

**O：把公司业务条线的合规预审从"经验依赖"升级为"知识库驱动"**

| KR | 衡量指标 | 目标值 |
|---|---|---|
| KR1 | 客户经理预审 TAT（Turn-Around Time） | ≤ 5 秒 |
| KR2 | 单笔预审规则覆盖率 | ≥ 92% |
| KR3 | 因规则遗漏导致的二次返工率 | ≤ 3% |
| KR4 | 合规官人工抽查通过率 | ≥ 95% |
| KR5 | 月活用户（客户经理 + 审批官） | ≥ 800 人 |

### 2.3 不做什么（Out of Scope）

- ❌ 不替代正式审批（仅作预审参考）
- ❌ 不接管客户档案管理（仍走核心系统）
- ❌ 不直接落账、不发起放款
- ❌ 不处理零售个人贷款（本期仅公司业务）

## 4. 产品定位

### 4.1 Wiki-RAG vs 普通 RAG · 立标对比

| 能力维度 | 普通 RAG | **Wiki-RAG（本产品）** |
|---|---|---|
| 文档建模 | 扁平片段集合 | **命名空间 + 类目层级**（业务/政策/监管/法规 4 大类） |
| 切分策略 | 固定窗口滑动切分 | **语义感知**：按"第X条/第X章/(一)(二)"切分 |
| 检索方式 | 纯向量相似度 | **混合检索**（语义 α=0.6 + BM25 0.4）+ RRF 融合 |
| 知识关联 | 片段彼此独立 | **同义词桥接 + 跨文档引用 + 实体追踪** |
| 单查询召回 | 1 次查询 → top-k | **多查询并行**：原始 + 改写 2 + 8 维预设模板 |
| 引用溯源 | 无引用 | **每条结论必带文件名 + 原文关键句** |
| 对话记忆 | 无状态 | **5 层记忆**（窗口/摘要/实体/引用/偏好） |
| 知识更新 | 重建全量索引 | **增量入库**（SHA256 去重）+ 软删除 |
| 合规审计 | 无法追溯 | **全程留痕** |
| 权限管控 | 统一暴露 | **RBAC** 按文档分级 |

### 4.2 一句话区分
- **普通 RAG**：把文档塞进向量库，检索相似句子
- **Wiki-RAG**：把规则建模成知识网络，检索"有命名空间 / 有层级 / 有引用 / 有版本 / 有权限"的知识节点，并具备跨轮记忆

## 6. 详细功能需求

### 6.1 场景 1：信贷预审合规助手（P0）

#### 6.1.1 功能描述
客户经理在表单中录入企业 + 业务要素，系统并行启动 8 个维度的 RAG 检索，2-5 秒内生成结构化预审报告。

#### 6.1.2 输入字段（13 项）
| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| 企业名称 | 文本 | ✅ | |
| 成立年份 | 数字 | ✅ | 用于推算经营年限 |
| 所属行业 | 文本 | ✅ | 触发行业准入查询 |
| 企业类型 | 枚举 | ✅ | 小微/中型/大型 |
| 年营业收入(万) | 数字 | ✅ | |
| 资产负债率(%) | 数字 | ✅ | 触发财务准入查询 |
| 内部评级 | 枚举 | ✅ | AAA/AA/A/BBB/BB |
| 不良记录 | 布尔 | ✅ | |
| 产品类型 | 枚举 | ✅ | 流贷/固贷/并购/票据/ABS |
| 申请金额(万) | 数字 | ✅ | 触发权限查询 |
| 期限(月) | 数字 | ✅ | |
| 贷款用途 | 文本 | ✅ | 触发用途合规检查 |
| 担保方式 + 估值 | 复合 | ✅ | 触发质押率检查 |

#### 6.1.3 处理流程
```
1. 构造 8 维检索查询（RETRIEVAL_QUERIES 模板填充）
2. 每条查询经 QueryRewriter 派生 2 个变体 → 24 次检索
3. 并行执行混合检索（ChromaDB + SQLite FTS5）
4. 按 chunk_id 合并去重，取 max(score)，保留 top-12
5. 拼装 system prompt + 申请描述 + 检索内容
6. Claude Sonnet 4.6 流式生成 8 段式报告
7. 解析结论（PASS/CONDITIONAL/REJECT），抽取来源文件
8. 渲染：检索动画 + 命中文件表 + 报告 + Verdict 卡片
```

#### 6.1.4 输出格式（8 段式报告）
1. 一、基本准入审查
2. 二、额度与期限合规性
3. 三、担保合规性
4. 四、定价建议
5. 五、审批权限建议
6. 六、风险提示
7. 七、预审结论（PASS/CONDITIONAL/REJECT）
8. 八、所需材料清单

每条结论格式：`✅/⚠️/❌ [条件名]：[判断]（依据：[文件名] - [原文关键句]）`

#### 6.1.5 性能要求
- 端到端 ≤ 5 秒（P95）
- 单笔命中 ≥ 5 份规则文档
- 引用召回率 ≥ 95%

### 6.2 场景 2：智能咨询多轮对话（P0）

#### 6.2.1 功能描述
客户经理可以连续提问，系统通过 5 层记忆机制实现指代消解、上下文延续、实体追踪。

#### 6.2.2 5 层记忆机制

| 层级 | 范围 | 触发条件 | 实现 |
|---|---|---|---|
| **① 短期窗口** | 最近 10 轮原文 | 始终启用 | sliding window |
| **② 长期摘要** | > 10 轮的对话压缩 | 触发条件 turn ≥ 20 | LLM summarize → store |
| **③ 实体追踪** | 企业名 / 金额 / 产品等 | 始终启用 | NER + persistent kv |
| **④ 引用记忆** | 已被引用的文档列表 | 始终启用 | doc_id set |
| **⑤ 偏好记忆** | 角色身份 / 详略风格 | 用户主动声明 | user_pref dict |

#### 6.2.3 关键功能点
- 指代消解："那 800 万呢" → 自动补全企业类型
- 接续问答："那利率怎么算" → 复用上文实体
- 摘要提示：「请汇总一下」→ 调用所有引用记忆
- 实时可视化：右侧记忆面板展示当前 5 层状态

#### 6.2.4 接口
- `POST /api/chat` { session_id, query }
- `GET /api/chat/memory/{session_id}` 返回当前记忆状态
- `DELETE /api/chat/{session_id}` 清空会话

### 6.3 场景 3：产品方案智能比对（P1）

#### 6.3.1 功能描述
用户输入业务诉求（如"1000万应收账款融资 期限 6 月"），系统并行检索多个候选产品 → 横向构建对比矩阵 → 推荐最优方案。

#### 6.3.2 处理流程
```
1. LLM 解析诉求 → 提取关键信号（场景/期限/金额/担保/期望）
2. 按关键信号匹配产品候选池（保理/质押贷/e销贷/票据贴现/...）
3. 并行检索每个产品的 8 维要素（额度/期限/利率/审批/材料/...）
4. 按场景适配度打分，标注每行最优值
5. LLM 生成综合推荐 + 引用溯源
```

#### 6.3.3 对比维度（8 列）
融资额度 / 融资期限 / 利率 / 审批层级 / 所需材料 / 行内权限 / 适用条件 / 风险点

### 6.4 知识库管理（P0）

#### 6.4.1 知识库总览页
- 4 项关键统计：文档总数 / 文本片段 / 向量索引 / 索引大小
- 文档清单按 4 大类目分组：业务规则 / 内部政策 / 监管文件 / 法律法规
- 每份文档显示：chunk 数 / 章节数 / 关键主题

#### 6.4.2 摄入流水线

**6 步工序（可视化流程）**

| 步骤 | 模块 | 技术实现 | 输出 |
|------|------|----------|------|
| ① 扫描加载 | DocumentLoader | 扫描 `data/raw/` 目录，识别 5 种格式 | 文件列表 + SHA256 |
| ② 格式解析 | DocumentParser | pdfplumber / python-docx / pandas 抽取文本+表格 | 原始文本 + 结构化数据 |
| ③ 数据清洗 | DataCleaner | 正则去噪、归一化、去页眉页脚 | 清洗后文本 |
| ④ 语义分块 | TextChunker | 按"第X条/第X章"切分，滑窗 fallback | 500字符/块，80字符重叠 |
| ⑤ 向量化 | Embedder | sentence-transformers MiniLM-L12-v2 | 384维向量，batch=32 |
| ⑥ 双库存储 | MetaStore + VectorStore | ChromaDB(向量) + SQLite FTS5(关键词) | 双重索引持久化 |

**功能特性**
- 支持格式：PDF / DOCX / MD / XLSX / CSV
- 增量摄入：SHA256 去重，已索引文档自动跳过
- 错误处理：解析失败/空文档标记 error 状态，不阻塞其他文档
- **可视化演示**：Web 界面提供「模拟摄入」按钮，逐文档动画展示 6 步处理过程，实时更新统计面板

### 6.5 RAG 知识库日志（P0 · 新增）

#### 6.5.1 功能描述
对所有知识库文件进行全生命周期管理：入库 / 审核 / 生效 / 暂停 / 废止 / 临时覆盖。

#### 6.5.2 文件状态机

```
            ┌──────[审核拒绝]──→[退回]
            │
[草稿]──→[审核中]──→[生效中]──┬──→[已废止]（被新版替代 / 主动废止）
                                │
                                └──→[暂停使用]（被临时文件覆盖）
                                       │
                                       └──→[恢复生效]（临时文件到期）
```

#### 6.5.3 元数据字段（每份文档必填）
| 字段 | 类型 | 说明 |
|---|---|---|
| 文件名 | str | |
| 类型 | enum | 常规 / 临时 |
| 分类 | enum | 业务规则 / 内部政策 / 监管文件 / 法律法规 |
| 状态 | enum | 生效 / 暂停 / 废止 |
| 入库时间 | datetime | |
| 发布部门 | str | |
| 发布人 | str | |
| 审核状态 | enum | 通过 / 拒绝 |
| 审核人 | str | |
| 审核时间 | datetime | |
| 生效起 | date | |
| 生效止 | date / null | null 表示长期有效 |
| 废止日期 | date / null | |
| 覆盖文件 ID | list | 仅临时文件需要 |
| 被覆盖标记 | bool | 常规文件被临时文件覆盖时为 true |

#### 6.5.4 临时文件 → 常规文件覆盖机制（关键）

**业务规则**：
1. 临时文件入库时声明 `覆盖文件 ID`
2. 系统将被覆盖的常规文件状态置为「暂停使用」
3. 检索时优先返回临时文件片段
4. 临时文件到期（定时任务每天 00:00 检查）：
   - 状态 → 已废止，废止日期 = 到期日期
   - 同步将被覆盖的常规文件恢复「生效」
5. 全程留痕，操作记录写入变更日志

**示例**：
> 《阶段性提高小微企业流贷单户限额的临时通知（2026年第3号）》
> · 类型：临时

## 8. 技术架构

### 8.1 技术选型（9 个核心组件）

| 组件 | 选型 | 关键参数 |
|---|---|---|
| LLM 推理 | Claude Sonnet 4.6 | max_tokens=2500, temperature=0.0 |
| Embedding | sentence-transformers MiniLM | dim=384, batch=32, normalize=True |
| 向量数据库 | ChromaDB | collection=credit_knowledge, cosine |
| 关键词检索 | SQLite FTS5 + jieba | k1=1.2, b=0.75 |
| 查询改写 | Claude few-shot | variants=2, max_tokens=200 |
| 混合检索 | RRF Hybrid | alpha=0.6, top_k=6, rrf_k=60 |
| 语义分块 | Custom Chunker | size=500, overlap=80, 5 种正则 |
| 文档解析 | pdfplumber / docx / pandas | 5 种格式路由 |
| 服务层 | FastAPI + Typer | uvicorn 异步流式 + CORS |

### 8.2 数据流（端到端）

```
┌──────────────┐
│  原始文档     │ PDF/DOCX/MD/XLSX
└──────┬───────┘
       ▼
┌──────────────┐  ┌──────────────┐
│ DocumentLoader│──│ MetaStore    │ 注册 doc_id, sha256, 元数据
└──────┬───────┘  └──────────────┘
       ▼
┌──────────────┐
│ DocumentParser│ → 抽取 raw_text + 表格
└──────┬───────┘
       ▼
┌──────────────┐
│ DataCleaner  │ → 去噪 / 归一 / 去重
└──────┬───────┘
       ▼
┌──────────────┐
│ TextChunker  │ → 按"第X条"切 + 滑窗 fallback
└──────┬───────┘
       ▼
┌──────────────┐  ┌──────────────┐
│  Embedder    │─▶│  ChromaDB    │ 向量索引
└──────┬───────┘  └──────────────┘
       ▼
┌──────────────┐
│ MetaStore    │─▶ SQLite FTS5（关键词索引）
└──────────────┘

       ↑↑↑ 离线摄入 ↑↑↑
─────────────────────────────────────
       ↓↓↓ 在线检索 ↓↓↓

┌──────────────┐
│  用户查询     │
└──────┬───────┘
       ▼
┌──────────────┐
│ QueryRewriter│ → 原始 + 2~3 变体
└──────┬───────┘
       ▼
┌──────────────┐
│HybridRetriever│ 语义×0.6 + BM25×0.4
└──────┬───────┘
       ▼
┌──────────────┐
│ MemoryManager│ 注入短期/实体/引用/偏好
└──────┬───────┘
       ▼
┌──────────────┐
│  Claude LLM  │ 流式生成 + 强约束引用
└──────┬───────┘
       ▼
┌──────────────┐
│  报告渲染     │ 引用高亮 + 来源链接
└──────────────┘
```

### 8.3 关键参数表

| 参数 | 取值 | 调优依据 |
|---|---|---|
| chunk_size | 500 | 覆盖完整规则条款 |
| chunk_overlap | 80 | 16% 重叠保留上下文 |
| embedding_dim | 384 | 多语种 MiniLM 平衡 |
| top_k | 6 | 覆盖率 92% |
| hybrid_alpha | 0.6 | 语义为主，关键词辅助 |
| rerank_top_n | 12 | 多查询合并后保留 |
| score_threshold | 0.35 | 低于视为不相关 |
| query_variants | 2~3 | 召回率 +15~20% |
| memory_window | 10 轮 | 短期窗口大小 |
| summary_trigger | 20 轮 | 触发摘要压缩 |
| llm_max_tokens | 2500 | 报告平均 1800 token |
| llm_temperature | 0.0 | 合规场景要求确定性 |

## 10. UI/UX 规范

### 10.1 视觉令牌

| Token | 值 | 用途 |
|---|---|---|
| primary | `#1E3A5F` | 主色（深银行蓝） |
| primary-dark | `#152D4A` | 深色辅 |
| accent-gold | `#C9A84C` | 金色强调 |
| success | `#16A34A` | 通过 / 生效 |
| warning | `#D97706` | 暂停 / 临时 |
| danger | `#DC2626` | 废止 / 不通过 |
| info | `#0284C7` | 提示 |
| purple | `#7C3AED` | 内部政策标识 |
| 字体 | Noto Sans SC + Inter | 中英文混排 |

### 10.2 交互原则

1. **即点即响**：核心动作 ≤ 1 次点击触发
2. **过程可见**：检索 / 推理过程实时动画展示，不让用户对着 loading 等
3. **引用可点**：每条引用都可点击展开原文（M3 阶段）
4. **状态明确**：用颜色 + 图标 + 文字三重提示状态（生效/暂停/废止）
5. **可逆操作**：所有删除 / 废止操作可在 30 天内回滚

### 10.3 关键视觉处理

- **被废止文件**：文件名 `text-decoration: line-through; color: #DC2626;`
- **被暂停文件**：文件名删除线 + 黄色背景行 + "暂停使用" pill
- **临时文件**：金色边框 + 进度条倒计时
- **生效中**：绿色 pill + 文字状态
