Wiki-RAG vs 普通 RAG · 立标对比
本产品采用了哪些 Wiki 化能力，与普通 RAG 的本质区别
本产品采用的 Wiki 化能力（Wiki-RAG ≠ 普通 RAG）
普通 RAG 把文档当作"扁平的文本片段集合"。Wiki-RAG 把文档视为有结构、有关系、有版本的知识网络 —— 每条规则有 命名空间（业务规则 / 内部政策 / 监管文件 / 法律法规）、有引用关系（"详见第X条"）、有层级标题（章/节/条/款）、有版本演进（旧规作废 / 新规生效）， 就像一个银行内部的 Wikipedia。检索的不是"片段"，而是"知识节点"。

① 命名空间 / 类目
10 份文档按"业务规则 / 内部政策 / 监管文件 / 法律法规"四大命名空间组织，检索可按 category 过滤。
类似 Wikipedia 的 Categories，但面向银行场景
② 层级标题感知
分块器识别"第X章/第X条/(一)(二)"等结构标记，每个 chunk 携带其所属章节信息。
检索时可还原"出自《准入标准》§5.1"
③ 跨文档引用
规则文档常含"详见《XX管理办法》"。Wiki-RAG 把这种引用解析为可跳转的内链。
类似 Wiki 的 [[wikilink]] 双向链接
④ 版本与有效性
每篇文档登记发文时间 / 生效日期 / 失效日期，过期规则不参与检索。
解决 LLM 不知最新监管时效的问题
⑤ 同义词与术语桥
"小微 / 普惠 / 中小微"是同一回事；"LPR / 贷款基础利率"指向相同概念。LLM 改写器自动桥接。
QueryRewriter 内置银行术语词典
⑥ 角色化视图
客户经理 / 审批官 / 风控官看到的"知识范围 + 答案详略 + 引用风格"不同。
通过偏好记忆 + RBAC 实现
⑦ 增量编辑
单份文档可独立更新（按 doc_id 删除旧 chunk + 新 chunk 入库），无需重建全量索引。
类似 Wiki 的 page revision
⑧ 全程留痕审计
每次问答记录：查询 / 命中片段 / 来源文件 / 模型版本 / token 用量，对接行内合规系统。
类似 Wiki 的 page history
⑨ 多轮记忆机制
5 层记忆（窗口/摘要/实体/引用/偏好），让助手"记住"已经讨论过的实体和已引用的文档。
普通 RAG 一问一答无状态
一句话区分
普通 RAG：把文档"塞进"向量库，检索"相似的句子"。
Wiki-RAG（本产品）：把规则"建模成知识网络"，检索"有命名空间 / 有层级 / 有引用 / 有版本 / 有权限的知识节点"，并具备跨轮记忆。
RAG 设计思路 · 4 个核心创新
01
语义感知分块，不切碎条款
银行规则的最小语义单元是"第X条"。通用滑窗会把"小微企业贷款额度不得超过500万元"切成两段，导致检索时只命中"500万元"丢失主语。
优先按 第X条/第X章/(一)(二) 切分 → 句界对齐 → fallback 到滑窗(500/80)
02
混合检索，覆盖"模糊问"和"精确查"
客户经理问"小微企业额度上限多少"是模糊语义，但要查"LPR-30BP"是精确数字。纯向量检索召回 LPR 类条款分数偏低，纯 BM25 又泛化不足。
语义(α=0.6) + BM25(1-α=0.4) + RRF 融合 · 数字/单位/英文缩写自动加权
03
多查询并行，单次预审命中 5+ 文档
一笔预审涉及"准入/期限/担保/定价/审批/行业/财务"等多个维度。仅靠用户一句话难以同时召回所有相关规则，导致漏审。
8 条预设查询模板 (RETRIEVAL_QUERIES) + LLM 改写 2 条 → 并行检索 → 合并去重
04
强引用约束，杜绝模型幻觉
合规场景容不得 LLM "一本正经胡说"。任何一句没出处的判断，都是审计漏洞。
System prompt 三重约束：以检索内容为唯一依据 + 每条结论必须括号注明文件名 + 知识库无规定时如实说明
